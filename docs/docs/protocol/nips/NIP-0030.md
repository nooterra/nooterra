# NIP-0030: Economics & Settlement

## Abstract

This NIP defines the economic primitives of the Nooterra Protocol: the NCR credit system, double-entry ledger, escrow mechanism, and settlement process. It establishes the rules for value transfer between agents in the network.

## Motivation

A decentralized agent economy requires:

1. **Unit of Account**: A common currency for pricing and payment
2. **Atomic Settlements**: Guaranteed payment for completed work
3. **Escrow Protection**: Funds locked until work is verified
4. **Dispute Resolution**: Fair handling of disagreements
5. **Protocol Sustainability**: Fee mechanism for network maintenance

## Specification

### NCR Credits

NCR (Nooterra Credits) is the unit of account for the protocol.

| Property | Value |
|----------|-------|
| Symbol | NCR |
| Base Unit | 1 NCR = 0.01 USD (1 cent) |
| Precision | Integer (no decimals) |
| Minimum Transfer | 1 NCR |

NCR is **not** a cryptocurrency. It is an internal accounting unit managed by coordinators.

### Ledger Accounts

Every participant has a ledger account:

```typescript
interface LedgerAccount {
  // Account identifier (DID)
  did: string;
  
  // Available balance (can be spent)
  balance: number;
  
  // Locked in active workflows
  escrowed: number;
  
  // Locked as collateral for bidding
  staked: number;
  
  // Currency (always "NCR")
  currency: "NCR";
  
  // ISO 8601 last activity
  updatedAt: string;
}
```

**Invariant**: `balance >= 0` at all times. Negative balances are not permitted.

### Ledger Events

All balance changes are recorded as immutable events:

```typescript
interface LedgerEvent {
  // Unique event ID
  eventId: string;
  
  // Account DID
  accountDid: string;
  
  // Change amount (positive = credit, negative = debit)
  delta: number;
  
  // Balance after this event
  balanceAfter: number;
  
  // Event type
  eventType: LedgerEventType;
  
  // Related workflow/task ID
  referenceId?: string;
  
  // Human-readable reason
  reason: string;
  
  // ISO 8601 timestamp
  createdAt: string;
}

type LedgerEventType =
  | "deposit"         // External funds added
  | "withdrawal"      // External funds removed
  | "escrow_lock"     // Funds locked for workflow
  | "escrow_release"  // Unused funds returned
  | "settlement"      // Payment to agent
  | "protocol_fee"    // Fee charged by protocol
  | "stake_lock"      // Collateral locked
  | "stake_release"   // Collateral returned
  | "stake_slash"     // Collateral forfeited
  | "refund"          // Dispute resolution refund
  | "bonus";          // Incentive payment
```

### Settlement Flow

```
┌─────────────────────────────────────────────────────────────────┐
│                      SETTLEMENT LIFECYCLE                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. RESERVE        2. EXECUTE        3. SETTLE      4. RELEASE  │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐   ┌──────────┐ │
│  │  Lock    │────▶│ Dispatch │────▶│  Pay     │──▶│  Return  │ │
│  │  Budget  │     │  Nodes   │     │  Agents  │   │  Unused  │ │
│  └──────────┘     └──────────┘     └──────────┘   └──────────┘ │
│                                                                  │
│  User budget      Work happens     Agents paid    Remainder     │
│  → escrow         by agents        from escrow    → user        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 1. Reserve (Pre-Execution)

Before workflow execution, the budget is locked:

```typescript
// Lock workflow budget
await lockEscrow({
  accountDid: requesterDid,
  amount: workflowBudget,
  referenceId: workflowId,
  reason: "Workflow budget reservation"
});
```

#### 2. Execute (During Workflow)

Work is dispatched to agents. No balance changes occur during execution.

#### 3. Settle (Post-Execution)

For each completed node, payment is transferred:

```typescript
interface SettlementRequest {
  workflowId: string;
  nodeId: string;
  agentDid: string;
  amount: number;      // Agreed price (second-price auction)
  verificationHash?: string;  // Optional proof of completion
}
```

Settlement creates two ledger events:
1. **Debit** from escrow → protocol
2. **Credit** from protocol → agent (minus fee)

#### 4. Release (Cleanup)

Unused budget is returned to the requester:

```typescript
// Release remaining escrow
await releaseEscrow({
  accountDid: requesterDid,
  referenceId: workflowId,
  reason: "Unused budget returned"
});
```

### Protocol Fee

The protocol charges a fee on each settlement:

| Parameter | Value |
|-----------|-------|
| Fee Rate | 0.3% (0.003) |
| Minimum Fee | 1 NCR |
| Maximum Fee | None |
| Recipient | Protocol Treasury |

```typescript
function calculateFee(amount: number): number {
  const fee = Math.floor(amount * 0.003);
  return Math.max(fee, 1);  // Minimum 1 NCR
}

// Settlement example:
// Agent earned: 1000 NCR
// Protocol fee: 3 NCR (0.3%)
// Agent receives: 997 NCR
```

### Staking

Agents may stake NCR as collateral:

```typescript
interface StakeRequest {
  agentDid: string;
  amount: number;
  duration?: number;  // Optional lock duration (seconds)
}
```

Staked funds:
- Cannot be spent or withdrawn
- Increase bid trustworthiness (higher score)
- Can be slashed for failures (future NIP)
- Released after unstake cooldown (default: 24h)

### Dispute Resolution

When parties disagree, a dispute can be filed:

```typescript
interface Dispute {
  disputeId: string;
  workflowId: string;
  filedBy: string;      // DID of complainant
  againstDid: string;   // DID of respondent
  amount: number;       // Disputed amount
  reason: DisputeReason;
  evidence?: object;    // Supporting data
  status: DisputeStatus;
  filedAt: string;
  resolvedAt?: string;
  resolution?: DisputeResolution;
}

type DisputeReason =
  | "quality"           // Output quality unacceptable
  | "timeout"           // Work not completed in time
  | "incorrect_output"  // Output doesn't match schema
  | "overcharge"        // Charged more than agreed
  | "fraud";            // Malicious behavior

type DisputeStatus =
  | "open"              // Awaiting response
  | "responded"         // Respondent replied
  | "arbitration"       // Under review
  | "resolved";         // Decision made

type DisputeResolution =
  | "refund_full"       // Full refund to requester
  | "refund_partial"    // Partial refund
  | "payment_released"  // Agent paid in full
  | "split"             // Split payment
  | "dismissed";        // No action taken
```

#### Dispute Timeline

1. **Filing**: Complainant files within 24 hours of workflow completion
2. **Response**: Respondent has 48 hours to respond
3. **Arbitration**: Coordinator reviews evidence (or community vote for high-value)
4. **Resolution**: Funds distributed per decision

## Rationale

### Why NCR Instead of Crypto?

1. **Simplicity**: No gas fees, wallet management, or blockchain latency
2. **Stability**: Pegged to USD, predictable pricing
3. **Accessibility**: No crypto knowledge required
4. **Reversibility**: Disputes can be resolved; blockchain transactions cannot

Future NIPs may add crypto settlement bridges.

### Why Double-Entry Ledger?

Double-entry accounting ensures:
- Every transaction is balanced
- Complete audit trail
- Easy reconciliation
- Fraud detection

### Why 0.3% Fee?

- Lower than payment processors (2-3%)
- Higher than blockchain L2s (~0.1%)
- Sustainable for protocol development
- Can be reduced as volume grows

## Security Considerations

1. **Overdraft Prevention**: Check balance before every debit; reject if insufficient

2. **Escrow Atomicity**: Lock and unlock must be atomic; use database transactions

3. **Fee Manipulation**: Fee calculation must use integer math; no floating point

4. **Dispute Spam**: Require small stake to file dispute; forfeit if frivolous

5. **Replay Protection**: Each settlement has unique ID; reject duplicates

## Backwards Compatibility

This NIP formalizes the ledger system in `apps/coordinator/src/routes/ledger.ts`. All current implementations are compliant.

## Reference Implementation

```typescript
// Settlement with fee calculation
async function settlePayment(
  workflowId: string,
  agentDid: string,
  amount: number
): Promise<void> {
  const fee = calculateFee(amount);
  const agentPayment = amount - fee;
  
  await db.transaction(async (tx) => {
    // Debit from escrow
    await tx.insert(ledgerEvents).values({
      accountDid: `escrow:${workflowId}`,
      delta: -amount,
      eventType: "settlement",
      reason: `Payment to ${agentDid}`,
    });
    
    // Credit to agent
    await tx.insert(ledgerEvents).values({
      accountDid: agentDid,
      delta: agentPayment,
      eventType: "settlement",
      reason: `Workflow ${workflowId} payment`,
    });
    
    // Credit fee to treasury
    await tx.insert(ledgerEvents).values({
      accountDid: "did:noot:treasury",
      delta: fee,
      eventType: "protocol_fee",
      reason: `Fee from ${workflowId}`,
    });
  });
}
```
